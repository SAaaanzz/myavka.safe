## 1. ОБЗОР ПРОЕКТА

### 1.1 Что это
GarantSafe — платформа-посредник (эскроу-сервис), которая выступает гарантом безопасности при онлайн-сделках между незнакомыми людьми. Платформа удерживает средства покупателя на Kaspi до тех пор, пока продавец не выполнит свои обязательства, после чего средства переводятся продавцу.

Пользователи регистрируются исключительно через Telegram (с привязкой номера телефона). Все сделки проходят через чат на платформе, куда в любой момент можно вызвать администратора для помощи, изменения данных или разрешения спора.

### 1.2 Основные направления (стартовые 5 платформ)

| # | Платформа | Аккаунты (верификация!) | Услуги (без верификации) |
|---|---|---|---|
| 1 | **Mobile Legends** | Продажа аккаунтов | Донат алмазов, буст ранга, прокачка, Starlight Member |
| 2 | **Telegram** | Продажа каналов, групп, ботов, аккаунтов | Telegram Stars, Telegram Premium, накрутка подписчиков, реклама в каналах |
| 3 | **PUBG Mobile** | Продажа аккаунтов | UC (Unknown Cash), Royal Pass, донат, буст ранга |
| 4 | **League of Legends** | Продажа аккаунтов | RP (Riot Points), буст ранга, прокачка, скины |
| 5 | **Free Fire** | Продажа аккаунтов | Алмазы, донат, буст ранга, Elite Pass |

> **Важно:** Каталог будет расширяться — любой клиент может предложить добавить новую игру/платформу. Но на старте фокусируемся на этих 5.

### 1.2.1 Правило верификации

| Тип лота | Верификация продавца | Причина |
|---|---|---|
| **Продажа аккаунта** | ✅ Обязательна (паспорт/ID + селфи) | Аккаунт можно восстановить через поддержку игры после продажи — это самый частый вид мошенничества |
| **Услуги** (донат, буст, Stars, Premium и т.д.) | ❌ Не требуется | Услуга одноразовая, после выполнения не может быть «отменена» |
| **Покупатель** (любой тип) | ❌ Не требуется | Покупатель защищён эскроу — его деньги заморожены до подтверждения |

### 1.3 Модель монетизации

#### Комиссии с продаж

| Тип сделки | Комиссия | Минимум | Описание |
|---|---|---|---|
| **Стандартная** (продажа аккаунта) | **7%** от суммы | **500 ₸** | Сделка с верификацией продавца, без участия администратора |
| **С администратором** (продажа аккаунта + админ) | **8%** от суммы **+ 500 ₸** | **500 ₸** + 500 ₸ за админа | Администратор входит в чат, контролирует, может менять данные |
| **Услуги** (донат, буст, Stars и т.д.) | **7%** от суммы | **500 ₸** | Без верификации, автоматический эскроу |
| **Услуги с администратором** | **8%** от суммы **+ 500 ₸** | **500 ₸** + 500 ₸ за админа | С участием администратора |

**Примеры расчёта:**
- Продажа аккаунта ML за 10,000 ₸ → комиссия 7% = 700 ₸ → продавец получает 9,300 ₸
- Продажа аккаунта ML за 5,000 ₸ → комиссия 7% = 350 ₸ → но минимум 500 ₸ → продавец получает 4,500 ₸
- Продажа аккаунта с админом за 50,000 ₸ → 8% = 4,000 ₸ + 500 ₸ = 4,500 ₸ → продавец получает 45,500 ₸
- Донат 100 Stars за 2,000 ₸ → 7% = 140 ₸ → но минимум 500 ₸ → продавец получает 1,500 ₸

#### Дополнительные источники дохода

| Источник дохода | Описание | Ставка |
|---|---|---|
| Вызов администратора в чат | Фиксированная плата за вход админа | 500 ₸ |
| VIP-статус продавца | Пониженная комиссия (на 1%), приоритет в каталоге | 5,000-15,000 ₸/мес |
| Продвижение объявлений | Верхние позиции в каталоге | 500-3,000 ₸ |
| Реклама | Баннеры на сайте | Договорная |
| Партнёрская программа | Вознаграждение за рефералов | 3% от комиссии |

## 3. АВТОРИЗАЦИЯ ЧЕРЕЗ TELEGRAM

### 3.1 Почему только Telegram

- **Номер телефона виден**: при авторизации через Telegram мы получаем ID пользователя, имя, username и (если пользователь разрешает) номер телефона — это повышает доверие между сторонами сделки
- **Единая точка входа**: не нужно запоминать логины/пароли
- **Встроенная 2FA**: Telegram сам обеспечивает защиту аккаунта
- **Мгновенные уведомления**: через того же Telegram-бота
- **Аудитория**: целевая аудитория (Казахстан, СНГ) активно пользуется Telegram

### 3.2 Техническая реализация

#### Telegram Login Widget (на сайте)
```
Пользователь нажимает «Войти через Telegram»
    │
    ▼
Открывается Telegram Login Widget (от @BotFather)
    │
    ▼
Пользователь подтверждает вход в приложении Telegram
    │
    ▼
Telegram отправляет на наш callback URL:
{
  id: 123456789,           // Telegram user ID (уникальный, неизменяемый)
  first_name: "Иван",
  last_name: "Петров",     // может быть пустым
  username: "ivan_petrov",  // может быть пустым
  photo_url: "https://t.me/i/userpic/...",
  auth_date: 1700000000,
  hash: "abc123..."        // подпись для проверки подлинности
}
    │
    ▼
Сервер проверяет hash (HMAC-SHA-256 с токеном бота)
    │
    ▼
Сервер ищет пользователя по telegram_id в БД
    │
    ├── Найден → Создаём сессию (JWT), пускаем на сайт
    │
    └── Не найден → Создаём нового пользователя:
                     │
                     ▼
                  Страница «Завершение регистрации»
                  ├── Подтверждение номера телефона (через Telegram-бота)
                  ├── Принятие оферты/правил
                  └── Готово → Создаём сессию, пускаем на сайт
```

#### Подтверждение номера телефона
```
При регистрации:
    │
    ▼
Пользователю в Telegram-бот приходит сообщение:
"Для завершения регистрации на GarantSafe, пожалуйста, 
 поделитесь своим номером телефона нажав кнопку ниже."
[📱 Поделиться номером телефона]   ← Telegram кнопка (request_contact)
    │
    ▼
Пользователь нажимает → бот получает объект Contact:
{
  phone_number: "+77001234567",
  first_name: "Иван",
  user_id: 123456789
}
    │
    ▼
Номер сохраняется в БД, привязывается к аккаунту
    │
    ▼
Регистрация завершена
```

### 3.3 Что видно другим пользователям при сделке

При создании сделки каждая сторона видит о контрагенте:

| Поле | Видимость | Пример |
|---|---|---|
| Telegram имя | Всегда | Иван П. |
| Telegram username | Если указан | @ivan_petrov |
| Номер телефона (маскированный) | Всегда | +7 700 *** ** 67 |
| Полный номер телефона | Только после оплаты сделки | +7 700 123 45 67 |
| Рейтинг | Всегда | ⭐ 4.8 (124 сделки) |
| Дата регистрации | Всегда | На платформе с 15.03.2025 |
| Статус верификации | Всегда | ✓ Верифицирован |
| Текущий IP (город) | Только администратору | Алматы, KZ |

### 3.4 Безопасность авторизации

| Мера | Описание |
|---|---|
| Проверка hash | Каждый логин проверяется через HMAC-SHA-256 с bot token |
| auth_date проверка | Логин-запрос действителен не более 5 минут (защита от replay attack) |
| Привязка по telegram_id | Уникальный, неизменяемый ID (не username, который можно сменить) |
| IP-логирование | Каждый вход записывает точный IP-адрес (см. раздел 7) |
| Rate Limiting | Не более 5 попыток входа за 10 минут с одного IP |
| Session Management | JWT Access token (15 мин) + Refresh token (30 дней) в httpOnly cookie |
| Автоматический выход | При смене IP на сильно отличающийся (другая страна) — принудительный re-login |

## 4. КАТАЛОГ ИГР И СЕРВИСОВ

### 4.1 Принцип каталога

**Любой клиент может создать свой лот (объявление)** — это открытый маркетплейс в стиле FunPay. Каждый лот привязан к конкретной игре/платформе и типу (аккаунт или услуга).

**Ключевое правило:**
- **Продажа аккаунта** → продавец обязан пройти **верификацию** (паспорт/ID + селфи), иначе лот не будет опубликован. Причина: аккаунт можно восстановить через поддержку игры после продажи.
- **Услуги** (донат, буст, Stars, Premium и т.д.) → верификация **не нужна**, лот публикуется после проверки администратором контента.

```
Продавец хочет продать аккаунт
    │
    ▼

    │
    ▼
Загрузка документов:
├── Фото документа (паспорт / ID-карта / водительское)
│   → Чёткое фото лицевой стороны
├── Селфи с документом в руке

    │

├── Антифрод-администратор проверяет:
│   ├── Документ настоящий (не фотошоп)
│   ├── Лицо совпадает с фото на документе
│   ├── Имя совпадает с Telegram-профилем (примерно)
│   └── Документ не в чёрном списке
│

│   → Значок 🔒✓ рядом с именем
│   → Может создавать лоты аккаунтов
│   → Документы хранятся зашифрованно (AES-256)
│
└── ❌ Отклонено → Причина отказа
    → Можно подать повторно через 24 часа

Сроки: обычно 1-12 часов (в рабочие часы — до 2 часов)
```

### 4.7 Зачем верификация для аккаунтов

**Проблема:** Продавец продаёт аккаунт игры через гаранта, получает деньги. Потом обращается в поддержку игры: «Мой аккаунт украли, помогите восстановить». Поддержка возвращает аккаунт исходному владельцу. Покупатель теряет и деньги, и аккаунт.

**Решение:** Верификация по документам означает:
1. Мы знаем реальную личность продавца
2. При мошенничестве — добавляем в чёрный список с реальными данными
3. Можно подать жалобу в правоохранительные органы
4. Сам факт верификации отпугивает мошенников
5. Покупатель видит значок 🔒 и чувствует себя безопаснее

### 4.8 Модерация каталога

- Каждый новый лот попадает в очередь модерации
- Администратор контента проверяет: запрещённый контент, соответствие категории, адекватность цены
- **Для аккаунтов**: дополнительно проверяется наличие верификации у продавца
- Автоматическая проверка текста на запрещённые слова (фильтр)
- Дублирование — автоматическое определение одинаковых лотов
- Если продавец получает 3+ жалобы — лот автоматически снимается
- Лоты аккаунтов, по которым были споры о восстановлении — автоматический красный флаг

## 7. IP-БЕЗОПАСНОСТЬ И АВТОМАТИЧЕСКАЯ ПРИОСТАНОВКА

### 7.1 Запись IP-адресов

**Принцип:** Каждое действие пользователя на платформе сопровождается записью точного IP-адреса. Все IP хранятся в отдельной таблице с полной историей.

**Когда записывается IP:**

| Событие | Записывается |
|---|---|
| Вход на сайт (авторизация через Telegram) | IP, User-Agent, timestamp |
| Открытие страницы (каждый запрос к API) | IP, endpoint, timestamp |
| Создание сделки | IP, deal_id, timestamp |
| Принятие сделки | IP, deal_id, timestamp |
| Отправка сообщения в чат | IP, deal_id, message_id, timestamp |
| Внесение оплаты | IP, deal_id, transaction_id, timestamp |
| Подтверждение получения | IP, deal_id, timestamp |
| Открытие спора | IP, deal_id, timestamp |
| Вызов администратора | IP, deal_id, timestamp |

### 7.3 Определение «резкой смены IP»

**Алгоритм проверки (при каждом действии в рамках активной сделки):**

```
При каждом запросе пользователя в рамках активной сделки:
    │
    ▼
Получаем текущий IP пользователя
    │
    ▼
Получаем «базовый IP сделки» — IP, с которого пользователь
создал/принял/оплатил эту сделку (первое значимое действие)
    │
    ▼

    │
    ├── IP совпадает полностью → ОК, продолжаем
    │
    ├── IP в той же подсети (/24) → ОК (динамический IP провайдера)
    │   Пример: было 185.45.67.123, стало 185.45.67.200 → ОК
    │
    ├── IP в том же городе/ISP, но другая подсеть → ПРЕДУПРЕЖДЕНИЕ
    │   Записываем, уведомляем, но не блокируем
    │   Пример: было 185.45.67.123, стало 185.45.90.55 → ⚠️

    └── IP в другом городе / стране / другой ISP → 🛑 ПРИОСТАНОВКА!
        Пример: было 185.45.67.123 (Алматы), стало 91.215.40.20 (Москва) → СТОП
```

### 7.5 Уровни IP-угрозы

| Уровень | Условие | Действие |
|---|---|---|
| 🟢 **Норма** | IP совпадает или в той же /24 подсети | Ничего |
| 🟡 **Внимание** | IP в том же городе/ISP, но другая подсеть | Запись в лог, лёгкое предупреждение (не блокировка) |
| 🟠 **Предупреждение** | IP в другом городе той же страны | Предупреждение пользователю, уведомление администратору |
| 🔴 **Приостановка** | IP в другой стране / кардинально другой сети / VPN/Tor | Мгновенная заморозка чата, вызов администратора |

### 7.6 Дополнительные IP-проверки

| Проверка | Описание | Действие |
|---|---|---|
| VPN/Proxy/Tor детект | Определение использования VPN через базы данных (ip2proxy, MaxMind) | При активной сделке → предупреждение + запись |
| Совпадение IP у обеих сторон | Покупатель и продавец с одного IP (самосделка?) | Автоматическая приостановка + уведомление администратору |
| IP из чёрного списка | IP ранее использовался заблокированным пользователем | Автоматическая блокировка входа + уведомление |
| Частая смена IP | Более 5 разных IP за час | Предупреждение + запись |
| Геолокация vs телефон | IP из другой страны, чем код телефона | Логирование, при сделке — предупреждение |

### 9.3 Распределение задач между администраторами

```
Новая задача (вызов, спор, IP-инцидент)
    │
    ▼
BullMQ очередь (Redis)
    │
    ▼
Диспетчер задач:
├── Определяет тип задачи (сделка / спор / IP / каталог)
├── Определяет приоритет (IP-инцидент = высший)
├── Находит свободных администраторов нужной специализации
├── Учитывает текущую нагрузку (макс. 10 активных дел)
├── Учитывает рейтинг администратора (для сложных задач → опытные)
└── Назначает ближайшему подходящему
    │
    ▼
Уведомление администратору:
├── Push в Telegram-бот: "⚡ Новая задача: [описание]"
├── Push на панели: звуковое уведомление + красный бейдж
└── Если не принято за 5 мин → переназначение другому
```

### 9.4 Графики работы (24/7)

| Смена | Время (UTC+6, Алматы) | Кол-во администраторов | Примечание |
|---|---|---|---|
| Утро | 08:00 — 16:00 | ~85 | Основной поток сделок |
| Вечер | 16:00 — 00:00 | ~85 | Пик активности |
| Ночь | 00:00 — 08:00 | ~30 | Пониженная нагрузка |

- Каждый администратор работает 8-часовую смену, 5 дней в неделю
- На выходные: дежурная смена ~40 администраторов
- При пиковых нагрузках (праздники, распродажи): усиленная смена

### 9.5 KPI администраторов

| Метрика | Цель | Как считается |
|---|---|---|
| Время первого ответа | < 15 мин | Время от поступления задачи до принятия |
| Время разрешения спора | < 4 часа | Время от открытия спора до решения |
| Обработанных сделок/день | 20-30 | Количество завершённых задач |
| Процент апелляций | < 10% | Кол-во обжалованных решений / все решения |
| Оценка пользователями | > 4.5/5 | Средняя оценка после завершения задачи |
| Время на IP-инцидент | < 10 мин | От уведомления до решения |

### 9.6 Мотивация и оплата администраторов

| Компонент оплаты | Описание |
|---|---|
| Фиксированная ставка | Базовая оплата за смену |
| Бонус за обработанные задачи | + за каждую успешно завершённую задачу |
| Бонус за высокий рейтинг | + при средней оценке > 4.7 |
| Штраф за пропущенные задачи | - при не принятии задачи в течение 15 мин |
| Премия месяца | Лучший администратор месяца (по KPI) |

| Device Fingerprint | Отпечаток устройства | hash123abc |

### 10.2 Автоматические проверки (при каждой сделке)

```
Пользователь создаёт/принимает сделку
    │
    ▼
Автоматическая проверка:
├── Telegram ID → в чёрном списке?
├── Номер телефона → в чёрном списке?
├── Текущий IP → в чёрном списке?
├── IP → VPN/Proxy/Tor?
├── Device Fingerprint → в чёрном списке?
├── Kaspi номер (если известен) → в чёрном списке?
├── Совпадение IP с другой стороной сделки?
├── Аккаунт создан < 24 часа назад + крупная сделка?
├── > 3 отменённых сделки за последние 7 дней?
└── Необычный паттерн поведения?
    │
    ├── Все проверки пройдены → Сделка продолжается
    │
    ├── 1-2 предупреждения → Сделка продолжается + запись в лог
    │                         + уведомление контрагенту
    │
    └── Блокирующий фактор → Сделка заблокирована
                              + уведомление администратору
                              + уведомление пользователю
```

### 12.2 Отдельный IP Security Service

Выделенный микросервис для IP-безопасности:

```
IP Security Service
├── Middleware (на API Gateway):
│   ├── Каждый запрос → извлечение IP из X-Forwarded-For / CF-Connecting-IP
│   ├── Запись в ip_logs таблицу
│   └── Если запрос в контексте сделки → проверка на смену IP
│
├── Endpoints:
│   ├── POST /ip/check — проверить IP (VPN, прокси, чёрный список)
│   ├── GET /ip/history/:user_id — история IP пользователя
│   ├── GET /ip/deal/:deal_id — все IP по конкретной сделке
│   └── POST /ip/alert — создать IP-инцидент
│
├── Workers (фоновые):
│   ├── IP Change Detector — сравнивает IP при каждом действии в сделке
│   ├── VPN/Proxy Detector — обогащает IP данными (MaxMind GeoIP2, ip2proxy)
│   └── Blacklist Checker — проверяет IP по чёрному списку
│
└── Внешние сервисы:
    ├── MaxMind GeoIP2 — геолокация и VPN-детект
    ├── ip2proxy — определение прокси/VPN/Tor
    └── AbuseIPDB — проверка репутации IP
```
