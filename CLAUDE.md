# CLAUDE.md — myavka.safe (GarantSafe)

Оркестрация разработки через сабагентов Claude Code. Этот файл читает главная
сессия (оркестратор) и все сабагенты. Держи его в актуальном состоянии.

---

## 1. Карта проекта

**Что это:** GarantSafe / Myavka.safe — эскроу-сервис (гарант) для безопасных
онлайн-сделок (игровые аккаунты и услуги: ML, PUBG, LoL, Free Fire, Telegram).
Регион — Казахстан/СНГ. Вход только через Telegram, оплата через Kaspi (в планах).
Полное ТЗ — `safe/PLAN_GARANTSAFE.md`, деплой — `safe/DEPLOY.md`.

**Стек:** Cloudflare Pages (статика) + Pages Functions (бэкенд) + отдельный
Worker с Durable Object (чат). **Сборки нет** — ни npm, ни бандлера, ни
TypeScript. Чистые HTML/CSS/ES-модули. Не добавляй зависимости/тулчейн без
явной просьбы пользователя.

```
safe/
├── site/                     # ФРОНТЕНД — билд-директория Pages (pages_build_output_dir="site")
│   ├── index.html catalog.html chats.html lots.html support.html blacklist.html faq.html
│   ├── css/style.css         # весь CSS, темы через :root[data-theme]
│   └── js/
│       ├── app.js            # i18n (RU/KZ/EN), тема, авторизация, промо-слайдер; общий для всех страниц
│       ├── catalog.js        # каталог/лоты (клиентские данные)
│       ├── chat.js           # живой чат: поллинг /api/chat, отправка, вложения
│       └── support.js        # форма/панель поддержки
├── functions/api/            # БЭКЕНД — Cloudflare Pages Functions (onRequestGet/onRequestPost)
│   ├── auth/[action].js      # tg-login, me  → сессии пользователей сайта
│   ├── chat/[room].js        # GET сообщений (?since), POST сообщения (+медиа) → пишет в DO
│   ├── deals/complete.js     # проверка проданного аккаунта по бан-базе, авто-бан продавца
│   ├── media/[id].js         # отдача медиа из KV
│   ├── support/[action].js   # логин сотрудников (PBKDF2), тикеты, управление персоналом
│   ├── telegram/webhook.js   # ИНТЕГРАЦИЯ: Telegram-бот (команды, вход, админ-действия)
│   └── health.js
├── workers/chat/             # ОТДЕЛЬНЫЙ Worker (свой деплой, свой wrangler.toml)
│   ├── src/index.js          # Durable Object ChatRoom: getMessages/addMessage, лимит 100 сообщений
│   └── wrangler.toml
└── wrangler.toml             # Pages-проект: KV "CHAT", привязка DO "CHAT_DO"→ChatRoom(script "myavka-chat-do")
```

**Хранилище — только Cloudflare KV (`env.CHAT`) и Durable Object (`env.CHAT_DO`).**
Отдельной SQL-БД нет. Каталог ключей KV (единый источник правды, не плоди
дубликаты):

| Префикс ключа | Значение | Где используется |
|---|---|---|
| `unick:<nick>` | `{tgId, balance, created, verified}` — аккаунт пользователя | auth, chat, telegram, deals |
| `session:<token>` | `nick` — сессия пользователя (token = 64 hex, TTL 30д) | auth, chat, deals, support |
| `login:<code>` | `{nick}` — одноразовый код входа (32 hex, TTL 300с) | telegram (выдаёт), auth (гасит) |
| `admin:<telegram_id>` | `{by, ts}` — администратор бота | telegram, deals (рассылка) |
| `ban:<nick>` | `{by, ts, ...}` — заблокированный пользователь | telegram, chat, deals |
| `banned_acc:<accountId>` | `{reason, ts}` — украденный/возвращённый аккаунт | telegram, deals |
| `media:<id>` | `{type, name, data}` — вложение (data-URI, TTL 30д) | chat (пишет), media (читает) |
| `ssession:<token>` | `login` — сессия сотрудника поддержки (TTL 12ч) | support |
| `staff:<login>` | `{salt, hash, dept, role}` — сотрудник (PBKDF2) | support |
| `ticket:<id>` | `{id, dept, from, subject, text, status, replies[], created}` | support |

**API (все под `/api/`, ответы — JSON):**

| Endpoint | Метод | Назначение |
|---|---|---|
| `/api/auth/tg-login` | POST | обмен `login`-кода на сессию |
| `/api/auth/me` | POST (Bearer) | текущий пользователь + баланс |
| `/api/chat/:room` | GET `?since=` / POST (Bearer) | сообщения комнаты / отправка |
| `/api/deals/complete` | POST (Bearer) | завершить сделку, проверить аккаунт по бан-базе |
| `/api/media/:id` | GET | отдать вложение |
| `/api/support/:action` | POST | login, ticket-create, me, tickets, ticket-update, create-staff… |
| `/api/telegram/webhook` | POST | приём апдейтов Telegram |
| `/api/health` | GET | healthcheck |

**Конвенции, которые НЕЛЬЗЯ ломать:**
- Bearer-токены пользователя валидируются регэкспом `^[0-9a-f]{64}$` до похода в KV.
- Флаг `verified` и любые привилегии берутся **только с сервера** (`unick:`),
  клиент не может их прислать. Не доверяй полям из тела запроса для прав доступа.
- Пароли сотрудников — **только PBKDF2** (см. `support/[action].js`), не хранить в открытом виде.
- Во фронтенде любой пользовательский текст экранируется `escHtml` перед вставкой в DOM.
- Права в боте: `OWNER_ID` (глава) > `admin:<id>` (администраторы). Не понижай проверки.
- Тестов в репозитории нет. Проверка = чтение кода + при необходимости `npx wrangler pages dev site` локально.

**Деплой (для контекста, не запускать без просьбы):** Pages-проект деплоится из
`site/` + `functions/`; Worker чата — отдельно `cd safe/workers/chat && npx wrangler deploy`.

## 2. Роль главной сессии — ОРКЕСТРАТОР (Opus 4.8)

Главная сессия **сама не делает всю работу**. Она:

1. **Разбирает** задачу пользователя на подзадачи по зонам (бэкенд / фронтенд / интеграции).
2. **Планирует** порядок: что параллельно, что последовательно.
3. **Делегирует** через инструмент `Agent`, давая каждому сабагенту полный контекст.
4. **Проверяет** результат через `reviewer` — цикл контроля качества (раздел 5) обязателен.
5. **Собирает** итог и отчитывается пользователю после каждого круга.

Оркестратор **сам** делает только мелочь: ответить на вопрос по коду, исправить
опечатку/однострочник, прочитать файл. Всё нетривиальное — через сабагентов.

## 3. Кто что делает: задача → агент → модель

| Задача | Агент | Модель | Инструменты |
|---|---|---|---|
| Разведка по коду перед планированием (где что лежит, имена, ключи KV) | `researcher` | haiku | Read, Grep, Glob (read-only) |
| Pages Functions / Worker / DO / `wrangler.toml` (`safe/functions/api/**` кроме telegram, `safe/workers/**`) | `backend-dev` | sonnet | Read, Edit, Write, Grep, Glob, Bash |
| Фронтенд (`safe/site/**`: html, css, js) | `frontend-dev` | sonnet | Read, Edit, Write, Grep, Glob, Bash |
| Telegram-бот, вход через Telegram, будущий Kaspi (`telegram/webhook.js`, `auth/[action].js`, оплата) | `integrations-dev` | sonnet | Read, Edit, Write, Grep, Glob, Bash |
| Ревью diff'а: баги, безопасность, соответствие задаче | `reviewer` | opus | Read, Grep, Glob, Bash (не пишет код) |
| Опечатка / вопрос по коду / тривиальный однострочник | оркестратор сам | — | — |

Границы зон описаны в `description` каждого агента и не должны пересекаться.
Задачу на стыке (например «поменять контракт API и подстроить фронт») дели на
подзадачи по зонам.

## 4. Правила делегирования

1. **Сначала разведка.** Перед раздачей работы разработчикам запусти
   `researcher` и получи точные пути/строки/имена. Сабагенты стартуют с нуля —
   не заставляй их гадать.
2. **Полный контекст в промпте.** Сабагент не видит историю чата. В задании
   всегда передавай: (а) исходную задачу пользователя дословно, (б) находки
   researcher'а (файл:строка), (в) конкретные ограничения из этого CLAUDE.md,
   (г) требуемый формат отчёта (что сделал + список файл:строка).
3. **Независимое — параллельно.** Если подзадачи не зависят по контракту
   (например правка бэкенда и несвязанная правка стилей) — вызывай нескольких
   сабагентов в одном сообщении.
4. **Зависимое — последовательно.** Если фронт зависит от нового API-контракта:
   сперва `backend-dev`/`integrations-dev` фиксирует контракт и возвращает его в
   отчёте, затем `frontend-dev` реализует под него.
5. **Не смешивай зоны.** backend-dev не трогает `safe/site/**`; frontend-dev не
   трогает `safe/functions/**` и `safe/workers/**`; правки Telegram/входа/оплаты —
   только integrations-dev.
6. **Мелочь — сам, без сабагента.**

## 5. Цикл контроля качества (ОБЯЗАТЕЛЕН)

Любая нетривиальная правка **не считается законченной, пока `reviewer` не выдал
`ВЕРДИКТ: PASS`**.

```
а) РАБОТНИК (backend/frontend/integrations-dev) делает правку и возвращает отчёт:
   что сделал + изменённые файлы со строками.
        │
        ▼
б) ОРКЕСТРАТОР запускает reviewer через Agent, передавая ему ДОСЛОВНО:
   — исходную задачу пользователя;
   — отчёт работника целиком.
        │
        ▼
в) reviewer возвращает вердикт:
   ┌── «ВЕРДИКТ: PASS»          → правка принята.
   └── «ВЕРДИКТ: НА ПЕРЕДЕЛКУ»  → оркестратор возвращает список находок
       ДОСЛОВНО ТОМУ ЖЕ работнику через SendMessage (НЕ спавнить нового
       агента — важно сохранить его контекст уже сделанной работы), затем
       снова шаг б).
        │
        ▼
г) Повторять до PASS, МАКСИМУМ 3 круга. Если после 3 кругов вердикт всё ещё
   «на переделку» — ОСТАНОВИТЬСЯ, не продолжать самому, вынести решение
   пользователю (описать, в чём разногласие и что предлагает reviewer).
        │
        ▼
д) После КАЖДОГО круга — краткий отчёт пользователю: кто делал (агент), что
   сделал, какой вердикт вынес reviewer.
```

**Механика инструментов:**
- Нового работника создаёшь через `Agent` (subagent_type = имя агента).
- На переделку возвращаешь его через `SendMessage` по его id/имени — контекст
  сохраняется, повторный `Agent` его бы обнулил.
- `reviewer` только находит проблемы и формулирует инструкции — **сам код не
  правит**. Правит всегда исходный работник.

## 6. Секреты

Токены и ключи (`TELEGRAM_BOT_TOKEN`, будущие `KASPI_*` и т.п.) — **только через
переменные окружения** (`env.*` в рантайме, задаются `wrangler pages secret put`
или в дашборде Cloudflare). **Никогда** не хардкодить секреты в коде, конфигах,
комментариях, фронтенде или коммитах. Любой сабагент, увидевший хардкод секрета,
обязан отметить это как проблему в отчёте/вердикте. (`OWNER_ID` в коде — это
публичный Telegram-ID главы, не секрет, но новые чувствительные значения —
только через env.)
